# Version 0.1.1 Real-Validation Specification

## Goal

Turn the public preview into something a new user can exercise with a real agent and evaluate against credible alternatives without overstating the evidence.

## Required behavior

### Codex CLI provider

- Accept the existing `AgentRequest` contract through `AgentProvider.invoke()`.
- Invoke `codex exec` without a shell and reuse the user's existing Codex authentication.
- Run ephemerally, in an empty temporary directory, with a read-only sandbox, outside the project, and pass `--ignore-user-config`.
- Select a role-specific JSON Schema with `--output-schema`.
- Parse JSONL events, returning the final agent message and reported token usage.
- Bound runtime and output size, terminate timed-out children, and preserve useful errors without exposing credentials.
- Support provider configuration through `{ "type": "codex-cli" }` with optional binary, model, timeout, and output limits.

### Validation cases

- Ship three sanitized, non-personal decision cases covering knowledge migration, open-source distribution, and local run storage.
- Give each case explicit criteria, constraints, a bounded configuration, and no embedded credentials or machine paths.
- Keep generated calls and runs out of the repository.

### Paired benchmark harness

- Compare `one_shot`, `sequential_grill`, and `decision_tree` observations for the same declared cases.
- Report blinded reviewer scores separately from inference cost.
- Report tokens, calls, latency, failures, and constraint violations.
- Classify whether each baseline is compute-matched to the treatment using a declared tolerance; never silently call an unmatched comparison matched.
- Preserve losses, ties, missing observations, and incomplete runs.
- Repeat the fixed evidence disclaimer that this harness does not establish universal superiority.

### User study kit

- Provide a five-participant protocol, onboarding timing sheet, scoring rubric, and results template.
- Keep uncollected human results visibly pending.

## Public test seams

1. `CodexCliProvider.invoke()` is the provider conformance seam.
2. `deliberate benchmark-compare` and its JSON/Markdown output are the benchmark seam.
3. `deliberate run` loading the packaged cases is the example-workflow seam.

## Acceptance

- Focused provider, benchmark, and case-validation tests pass.
- The complete existing test suite, typecheck, synthetic benchmark, documentation links, release audit, build, package inspection, and dependency audit pass.
- At least one real Codex provider smoke call succeeds when local authentication is available.
- Automated results are reported as automated results; five-user evidence is not claimed until five people actually participate.
