# Version 0.2.0 Product Decision Flow Specification

## Product promise

Turn one ambiguous product or engineering decision into a review-ready Decision Dossier: ask only high-leverage clarification questions, deliberate within a bounded budget, recommend one path, preserve alternatives and uncertainty, and wait for Human Approval.

## Audience

Product managers, engineering managers, tech leads, and staff engineers deciding what to build or how to design it.

## Default experience

1. The user enters a decision and optional context in one local web page.
2. The product reflects its current understanding and asks one question at a time.
3. A question is eligible only when its answer could materially change the recommendation, alternatives, criteria, constraints, or required evidence.
4. The interview stops when framing is sufficient or after four answered questions.
5. The product shows the framed decision and criteria before analysis begins.
6. Bounded deliberation runs through the existing deterministic controller.
7. The result opens as a decision page, not as a graph.
8. The user can inspect the reasoning tree or export a portable ADR-style Markdown document.

The normal flow exposes no agent counts, traversal policies, depth controls, or coverage terminology. The dossier must still identify partial evidence honestly.

## Decision page

The page must show:

- the recommendation and executive summary;
- score, confidence, and completeness;
- ranked alternatives and why they lost;
- assumptions, trade-offs, unresolved questions, and uncertainty;
- evidence with source and strength;
- branch, question, and evaluated-leaf counts;
- a link to the existing reasoning-tree viewer; and
- an ADR export link.

## Product workflow interface

The product module presents three operations:

1. begin a session from a decision statement and optional context;
2. answer the current clarification question; and
3. deliberate a ready session into a persisted Decision Dossier.

Only one unanswered question may exist at a time. Invalid transitions are rejected. Interview and framing judgments use the configured `AgentProvider`; deterministic code owns session state, question limits, request validation, run IDs, persistence, and the transition into the existing `DecisionEngine`.

## Local application

- `deliberate app --provider <provider.json>` starts the product on loopback.
- Existing `deliberate view` behavior remains read-only.
- Product writes are exposed only when a product workflow is explicitly configured.
- JSON request bodies are bounded and validated.
- Cross-site browser writes are rejected.
- Generated runs stay in the configured local run directory.

## Evidence boundary

The product recommends within the supplied context, admitted decision space, provider judgments, criteria, and budget. It does not claim exhaustive truth, execute the decision, or replace Human Approval.

## Public test seams

1. `DecisionProduct.begin()`, `.answer()`, and `.deliberate()` form the workflow seam.
2. The loopback HTTP flow forms the product experience seam.
3. `renderDecisionAdr()` forms the portable-artifact seam.
4. Existing viewer endpoints form the reasoning-navigation seam and must remain backward compatible.

## Acceptance

- A deterministic provider can complete intake, one clarification, framing, deliberation, ADR export, and viewer navigation end to end.
- A malformed interview or invalid transition fails without silently repairing state.
- The application contains no user-facing BFS, DFS, bootstrap, or agent-count configuration.
- Focused product tests, complete tests, typecheck, deterministic benchmark, documentation links, release audit, build, package inspection, and dependency audit pass.
